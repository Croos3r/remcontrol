# remcontrol

Control your PC mouse and keyboard from your phone. The phone acts as a
trackpad and keyboard; a Rust server on the PC injects the input events.

- `server/` — Rust WebSocket server (Linux/X11 and Windows)
- `app/` — Expo React Native app (TypeScript, Android)

## Quick start

### 1. Server (on the PC)

```sh
cd server
cargo run --release
```

The terminal prints the address, the pairing token, and a QR code.

- Linux needs an X11 session and `libxdo-dev` (Debian/Ubuntu) to build.
- Windows asks for firewall permission on first run: accept it.

See [server/README.md](server/README.md) for details.

### 2. App (on the phone)

The app uses native modules (`react-native-zeroconf`), so it needs a dev
build. Expo Go is not enough (QR scan and manual entry would work there, but
discovery would not).

```sh
cd app
npm install
npx expo run:android
```

With the phone plugged in (USB debugging on), this builds and installs the app.

### 3. Connect

Three ways, all on the same Wi-Fi/LAN as the PC:

- **Scan** the QR code from the server terminal
- **Discover**: pick the server from the mDNS list, enter the token once
- **Manual**: type IP, port (17890) and token

The app remembers the last successful connection and reconnects automatically
on launch.

## Gestures

| Gesture | Action |
|---|---|
| 1-finger pan | Move pointer |
| Tap | Left click |
| 2-finger tap | Right click |
| 2-finger pan | Scroll |
| Double-tap and hold | Drag |

The bottom toolbar has the keyboard toggle (opens the phone keyboard and types
on the PC), Esc/Tab/arrows/Enter keys, pointer speed presets, and disconnect.

## Security

LAN only. The server binds to your LAN IP by default (not `0.0.0.0`); override
with `bind_addr` in the config file.

Every connection runs a PSK-ECDH handshake: the client and server derive
ChaCha20-Poly1305 session keys from the pairing token and an ephemeral
X25519 exchange, so all trackpad and keyboard traffic is encrypted and
replay-protected. The token never travels in plaintext after pairing.

- The pairing token is generated by the server and shown only in the QR code.
  The terminal prints a short fingerprint, not the token itself.
- The phone stores the token in the OS keychain (SecureStore on Android), not
  in the app's plain AsyncStorage.
- The server rate-limits failed handshake attempts per IP and caps concurrent
  handshakes.
- WebSocket `Origin` is checked against an allowlist (empty by default rejects
  all browser origins; the native app sends none). Add yours to
  `allowed_origins` in the config if you build a web client.
- Rotate the token with `cargo run --release -- --reset-token`; previously
  paired phones must re-pair.

## Releases

Push a `v*` tag (e.g. `v1.0.0`) to trigger the release workflow. It builds the
server binaries (Linux, macOS Intel/ARM, Windows) and the Android APK, and
drafts a GitHub Release with all artifacts attached. The release is a draft:
review it and publish manually.

The macOS binaries are unsigned. On first run, clear the quarantine attribute:

```sh
xattr -dr com.apple.quarantine remcontrol-server-*
```

The Linux server needs `xdotool` at runtime for text entry.
